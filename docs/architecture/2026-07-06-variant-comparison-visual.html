제어 흐름은 누가 쥐는가

ReAct 자율 에이전트와 내가 만든 A · A-hub의 차이는 에이전트 개수가 아니라 “다음에 무엇을 할지”를 코드가 정하느냐 LLM이 정하느냐에 있다. 이 한 축이 예측가능성 · 지연 · 비용 · 안전을 전부 가른다 — 그리고 그 값을 측정으로 증명했다.

A — 코드(규칙)가 결정 · workflow A-hub — LLM이 라우팅만 결정 · 중간형 B — LLM이 도구를 자율 결정 · ReAct agent
개념 · ReAct

ReAct = Reasoning + Acting

Shunyu Yao 외, “ReAct: Synergizing Reasoning and Acting in Language Models” (ICLR 2023). 이전에는 따로 연구되던 두 가지를 한 궤적에서 교차시킨 것이 핵심이다.

Chain-of-Thought는 머릿속으로만 추론해 외부 세계에 접지되지 않고(→ 환각·지식 갱신 불가), Action-only는 도구는 부르지만 “왜”라는 추론 궤적이 없다. ReAct는 생각(Thought)이 계획을 세우고 갱신하며, 행동(Action)이 외부에서 새 정보를 끌어와 다시 추론에 주입하는 루프를 돌린다.

ReAct trajectory · LLM이 루프를 스스로 통제
Thought 1 답하려면 먼저 관련 근거를 검색해야 한다 Action 1 search("전자상거래 환불 기한") Observation 1 [검색 결과 5건] Thought 2 조문 원문이 빠졌으니 좁혀서 다시 검색 Action 2 get_law_article("전자상거래법", 17) Observation 2 [제17조 원문] Thought 3 근거가 충분하다 · 이제 답을 작성한다 Action 3 finish("...")

본질: 다음에 어떤 도구를 · 몇 번 부를지 · 언제 멈출지를 코드가 아니라 LLM이 매 스텝 스스로 정한다. 이것이 오늘날 “agentic” 시스템의 표준 골격 (LangGraph create_react_agent, function-calling loop)이 됐다.

요약 축

에이전트 개수 ≠ agentic

A는 에이전트 4개짜리 결정론 워크플로, B는 에이전트 1개짜리 자율 agent다. 이 역전이 핵심 — agentic의 척도는 개수가 아니라 LLM이 제어 흐름을 쥔 정도다.

자율성은 공짜가 아니다

A는 10.2s로 느리지만 안전·정확 · B는 6.4s로 빠르고 유연하나 안전 소폭 양보. agentic은 토큰·지연·실패모드(예: #68 컨텍스트 초과)를 늘린다.

현업 교훈

“에이전트가 결정하게 하자”는 유연성이 정말 필요할 때만 값어치가 있다. 분기가 자명한 워크플로에 LLM 조율을 넣으면 지연·비용·불안정성만 는다. 이는 Anthropic “Building Effective Agents”의 “대부분의 프로덕션은 workflow가 낫고, agent는 필요할 때만”과 정확히 일치하며 — 나는 이걸 말이 아니라 격리 측정으로 증명했다.

관점 · MAS

MAS의 옷을 입은 workflow

구조상 YES, 의사결정상 부분적으로만 YES. Hub-Spoke 슈퍼바이저 토폴로지와 역할 전문화는 교과서적 MAS다. 그러나 정통 MAS의 본질은 각 에이전트·슈퍼바이저가 LLM으로 자율 판단하는 것 — A는 조율 두뇌가 규칙(결정론)이다.

  • AMAS의 옷을 입은 workflow — 구조는 MAS, 판단은 결정론.
  • A-hub슈퍼바이저만 LLM화 — MAS에 한 발 더 가깝지만 도구 자율성은 없음.
  • B단일 에이전트인데 가장 agentic — 이 역전이 프로젝트가 측정하는 대비의 정수.
요약

한 장 결론

  • 01ReAct는 Thought↔Action↔Observation 루프를 LLM이 스스로 돌리는 패러다임 — 오늘날 agentic의 표준 골격.
  • 02A는 ReAct가 아니라 MAS 구조의 결정론 워크플로 · A-hub는 라우팅만 LLM화한 대조군 · B만 진짜 ReAct agent.
  • 03초기 설계의 ①②는 타당, ③은 과잉이었으나 측정 대상으로 남긴 것이 최고의 결정 — M8이 “LLM 조율 = 순수 오버헤드(+74%, 결정 100% 동일)”를 실증.
  • 04이 프로젝트의 가치는 “어느 하나가 최고”가 아니라 아키텍처 선택을 숫자로 비교·설명할 수 있다는 것.
