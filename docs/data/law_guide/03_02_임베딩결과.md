# 법령 및 가이드 데이터 임베딩 결과

## 1. 임베딩 전략

### 1.1 모델 선택
- **임베딩 모델**: OpenAI `text-embedding-3-large`
- **차원 축소**: 3072차원 → 1536차원 (MRL 기법 적용)
- **목표**: 높은 성능(large 모델)과 효율적 저장(small 모델 차원)의 균형

### 1.2 MRL (Matryoshka Representation Learning) 기법

#### MRL이란?
MRL은 임베딩 벡터를 계층적으로 구성하여, 벡터의 앞부분만 사용해도 전체 의미를 유지할 수 있도록 학습하는 기법입니다.

#### 적용 방식
```python
response = client.embeddings.create(
    model="text-embedding-3-large",
    input=texts,
    dimensions=1536  # 3072 → 1536 차원 축소
)
```

#### 장점
1. **성능 우수**: `text-embedding-3-large`의 높은 품질 유지
2. **효율적 저장**: 1536차원으로 저장 공간 50% 절감
3. **빠른 검색**: 낮은 차원으로 벡터 유사도 계산 속도 향상
4. **의미 보존**: MRL 학습 방식으로 차원 축소에도 의미 유지

### 1.3 파일별 처리 전략
- **출처별 독립 파일**: 각 법령/가이드를 별도 파일로 유지
- **이유**:
  - 출처별 독립적인 업데이트 가능
  - 특정 법령만 재처리 시 효율적
  - 병렬 처리 가능
  - 파일 손상 시 영향 범위 최소화

### 1.4 배치 처리
- **배치 크기**: 100 청크/배치
- **재시도 로직**: Rate limit 대응 (최대 3회, 지수 백오프)
- **진행률 추적**: tqdm 라이브러리로 실시간 진행 상황 표시

## 3. 임베딩 데이터 구조

### 3.1 JSONL 형식
각 줄은 독립적인 JSON 객체로 구성:

```jsonl
{"chunk_id": "...", "법령명": "...", "text": "...", "embedding": [...], ...}
```

### 3.2 추가된 필드
각 청크에 다음 필드가 추가되었습니다:

| 필드명 | 타입 | 설명 | 예시 |
|--------|------|------|------|
| `embedding` | List[float] | 1536차원 임베딩 벡터 | [0.056, -0.030, ...] |
| `embedding_model` | str | 사용된 임베딩 모델 | "text-embedding-3-large" |
| `embedding_dimensions` | int | 벡터 차원 수 | 1536 |
| `embedded_at` | str | 임베딩 생성 시각 (ISO 8601) | "2026-01-19T21:11:04.810016" |

### 3.3 원본 메타데이터 보존
청킹 단계에서 생성된 모든 메타데이터가 보존됩니다:
- `chunk_id`: 청크 고유 식별자
- `법령명`, `법령번호`, `시행일`, `법령유형`
- `조문번호`, `조문제목`, `hierarchy_path`
- `참조조문`, `준용조문`, `위임대상`, `keywords`
- 기타 모든 원본 필드

## 6. 다음 단계: PostgreSQL + pgvector 구축

### 6.1 DB 테이블 스키마 설계
```sql
CREATE TABLE law_embeddings (
    chunk_id TEXT PRIMARY KEY,
    법령명 TEXT,
    법령번호 TEXT,
    조문번호 TEXT,
    조문제목 TEXT,
    text TEXT,
    embedding VECTOR(1536),  -- pgvector 타입
    keywords TEXT[],
    hierarchy_path TEXT,
    -- 기타 메타데이터 필드들
    embedded_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT NOW()
);
```

### 6.2 pgvector 인덱스 생성
```sql
-- IVFFlat 인덱스 (빠른 근사 검색)
CREATE INDEX ON law_embeddings
USING ivfflat (embedding vector_cosine_ops)
WITH (lists = 100);

-- 또는 HNSW 인덱스 (더 정확한 근사 검색)
CREATE INDEX ON law_embeddings
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);
```

### 6.3 데이터 삽입 방법
1. **JSONL 파일 읽기**: Python으로 각 파일 파싱
2. **배치 삽입**: psycopg2 또는 asyncpg로 대량 삽입
3. **트랜잭션 관리**: 파일별 트랜잭션으로 무결성 보장

### 6.4 검색 쿼리 예시
```sql
-- 유사도 검색 (상위 5개)
SELECT
    chunk_id,
    법령명,
    조문번호,
    text,
    1 - (embedding <=> :query_embedding) AS similarity
FROM law_embeddings
ORDER BY embedding <=> :query_embedding
LIMIT 5;
```

### 6.5 권장 사항
1. **연결 풀링**: pgBouncer 또는 연결 풀 사용
2. **인덱스 튜닝**: 데이터 크기에 따라 lists/m 파라미터 조정
3. **하이브리드 검색**: 벡터 검색 + 키워드 검색 결합 고려
4. **모니터링**: 쿼리 성능 및 인덱스 사용률 추적
