# 행정규칙 청킹 전략 - 계층적 청킹

## 개요

**대상 파일**: 5개 Guide JSON 파일
**목표**: 사용자가 **사례를 제시했을 때 적절한 법률 기준을 찾아올 수 있도록** 예시 중심의 청킹 전략 수립

**핵심 전략**: **계층적 청킹 (Parent-Child Chunking)**
- **검색**: 작은 자식 청크로 정확도 향상
- **컨텍스트**: 큰 부모 청크로 문맥 완전 보존
- **장점**: 사례 검색 정확도 + LLM 답변 품질 동시 향상

## 1. 행정규칙 특성 분석

### 1.1 파일 목록 및 구조 유형

| 파일명 | 구조 유형 | 문단 수 | 예시 포함 | 청킹 전략 |
|--------|----------|---------|---------|---------|
| 전자상거래 등에서의 소비자보호 지침 | 계층형 | 228 | ✅ 많음 | **계층적 청킹** ⭐ |
| 특수판매에서의 소비자보호 지침 | 계층형 | 305 | ✅ 많음 | **계층적 청킹** ⭐ |
| 선불식 할부거래에서의 소비자보호 지침 | 계층형 | 393 | ⚠️ 중간 | 계층적 청킹 |
| 콘텐츠이용자 보호지침 | 조문형 | 151 | ❌ 없음 | 일반 청킹 |
| 소비자분쟁해결기준 | 조문형 | 4 | ❌ 없음 | 일반 청킹 |

**핵심 인사이트:**
- **전자상거래 지침**과 **특수판매 지침**이 예시가 풍부 → **사례 기반 검색의 핵심 파일**
- 계층형(3개)과 조문형(2개) 두 가지 스타일
- 예시 항목(`<예시>`, `ㅇ`, `설명`, `참고`) 구조가 중요
- **계층적 청킹**으로 정의 + 예시 완전 연결

## 2. 계층적 청킹이 필요한 이유

### 2.1 행정규칙의 특수성

#### 행정규칙 vs 법률/시행령 비교

| 특성 | 행정규칙 | 법률/시행령 | 최적 전략 |
|------|-----------|-----------|---------|
| **자기 완결성** | ❌ 예시만으로 불완전 | ✅ 조문만으로 완전 | 행정규칙: 계층적 청킹 |
| **정의 필요** | ✅ 정의 없으면 이해 어려움 | ❌ 조문 자체가 규범 | 법률: 일반 청킹 |
| **사례 의존** | ✅ 구체적 사례로 설명 | ❌ 추상적 법률 문언 | |
| **문맥 필요** | ✅ 관련 예시 비교 필요 | ❌ 단일 조문으로 충분 | |

#### 예시: 법률 조문 (자기 완결적)

```
민법 제110조 (사기, 강박에 의한 의사표시)
① 사기나 강박에 의한 의사표시는 취소할 수 있다.
```

✅ **완벽**: 이것만으로 충분한 법률 규정

#### 예시: 행정규칙 (의존적)

```
<예시 (1)>
갑은 인터넷 광고를 본 후 상점을 직접 방문하여 구입
⇒ 통신판매에 해당하지 않는다.
```

❌ **불완전**: 통신판매가 뭔데? 왜 해당 안 하는데? 다른 경우는?

### 2.2 사용자 시나리오 비교

#### 시나리오 1: 일반 청킹 (현재 문제)

**사용자 질문:**
> "인터넷 광고 보고 전화 주문하고 가게에서 받은 건 통신판매야?"

**일반 청킹 검색:**
```
자식 청크 (예시 1)만 검색됨:
"갑은 인터넷 광고를 본 후 상점을 직접 방문하여 구입
 ⇒ 통신판매에 해당하지 않는다."
```

**LLM 답변:**
> "통신판매에 해당하지 않습니다."

**문제점:**
- ❌ 왜? 설명 없음
- ❌ 통신판매가 뭔지 정의 없음
- ❌ 다른 사례와 비교 불가
- ❌ 법적 근거 없음

---

#### 시나리오 2: 계층적 청킹 (해결책)

**사용자 질문:**
> "인터넷 광고 보고 전화 주문하고 가게에서 받은 건 통신판매야?"

**계층적 청킹 검색:**
```
1. 자식 청크 (예시 1) 매칭 (정확도 높음)
2. 부모 청크 (전체) 가져오기:

【부모 청크】
가. 통신판매 정의
"통신판매"는 사업자의 판매에 관한 정보제공 및 소비자의 청약이
직접 대면하지 아니하고 이루어진 것을 의미한다.

<예시 (1)>
갑은 인터넷 광고를 본 후 상점을 직접 방문하여 구입
⇒ 청약이 비대면 아님 → 통신판매 아님

<예시 (2)>
사업자 C는 사이버몰에서 이용권 판매
⇒ 정보제공+청약 모두 비대면 → 통신판매 해당

<예시 (3)>
사업자 E는 해외 사이버몰 재화 대행 구매
⇒ 비대면 판매 → 통신판매 해당
```

**LLM 답변:**
> "통신판매는 정보제공과 청약이 모두 비대면으로 이루어진 경우입니다.
>
>  귀하의 경우:
>  - 인터넷 광고 (비대면) ✅
>  - 전화 주문 (비대면) ✅
>  - **가게에서 수령 (대면)** ❌
>
>  예시 (1)과 유사하게 청약이 대면으로 이루어졌으므로
>  **통신판매에 해당하지 않습니다.**
>
>  참고로 예시 (2)처럼 사이버몰에서 주문하고 배송받으면 통신판매입니다."

**장점:**
- ✅ 정의 포함 (통신판매가 뭔지)
- ✅ 판단 근거 (왜 해당 안 하는지)
- ✅ 사례 비교 (다른 경우는 어떤지)
- ✅ LLM이 깊이 있는 답변 가능

## 3. 계층적 청킹 전략

### 3.1 청킹 레벨 설계

```
부모 청크 (Parent Chunk)
├─ 소분류 전체 (가/나/다)
├─ 정의 + 모든 예시 포함
├─ 크기: 500-1000자
└─ 임베딩: ❌ 안 함 (검색 대상 아님)

자식 청크 (Child Chunk)
├─ 도입 본문 (정의/개념)
├─ 예시 (1), (2), (3)... (각각 개별)
├─ 크기: 100-300자
└─ 임베딩: ✅ 함 (검색 대상)
```

**관계:**
- 자식 → 부모: `parent_chunk_id` 참조
- 부모 → 자식: `자식_청크_ids` 배열 (선택)

---

### 3.2 부모 청크를 임베딩 안 하는 이유

#### 이유 1: 검색 정확도 (가장 중요!)

**부모 청크 임베딩의 문제:**
```
부모 청크 (4개 예시 혼합):
- 예시 (1): 직접 방문 → 통신판매 아님
- 예시 (2): 사이버몰 → 통신판매 해당
- 예시 (3): 해외 대행 → 통신판매 해당
- 예시 (4): 배송 용역 → 통신판매 해당

임베딩 벡터: [4개 예시 혼합]
→ 🔴 신호 희석 (Signal Dilution)
→ 🔴 "해당"과 "해당 안 함" 섞여서 매칭 점수 낮아짐
```

**자식 청크 임베딩:**
```
자식 청크 (예시 1만):
"인터넷 광고 → 직접 방문 → 통신판매 아님"

임베딩 벡터: [예시 1만 순수하게]
→ ✅ 매칭 점수 높음! 🎯
```

#### 이유 2: 중복 제거

**부모+자식 모두 임베딩하면:**
```
검색 결과:
1. 부모 청크 (전체) - 유사도 0.75
2. 자식 청크 (예시 1) - 유사도 0.85 ⭐
3. 자식 청크 (예시 2) - 유사도 0.60
4. 부모 청크 (다시) - 유사도 0.73

🔴 문제: Top-K 결과 중복, 혼란
```

**자식만 임베딩:**
```
검색 결과:
1. 자식 청크 (예시 1) - 유사도 0.92 🎯
2. 자식 청크 (예시 3) - 유사도 0.68
3. 다른 소분류 자식 - 유사도 0.65

✅ 명확하고 중복 없음
```

#### 이유 3: 비용 절감

**OpenAI 임베딩 비용:**
- text-embedding-3-large: $0.00013 per 1K tokens

**계산:**
```
부모 150개 × 800자 = 120,000자 ≈ 40,000 tokens
자식 500개 × 200자 = 100,000자 ≈ 33,000 tokens

부모+자식: 73,000 tokens ≈ $9.5
자식만: 33,000 tokens ≈ $4.3

절약: $5.2 (55%)
```

### 3.3 청킹 유형 결정 기준

| 구조 유형 | 기본 단위 | 예시 처리 | 부모 청크 | 자식 청크 |
|----------|----------|---------|---------|---------|
| **계층형 (예시 많음)** | 소분류(가/나/다) | **예시별 분할** | 소분류 전체 | 도입본문 + 예시별 |
| **계층형 (예시 적음)** | 소분류(가/나/다) | 본문 포함 | 중분류 단위 | 소분류별 |
| **조문형 (조-호)** | 조 단위 | 호 많으면 분할 | (사용 안 함) | 조 단위 |
| **조문형 (조-항-호)** | 조 단위 | 항 많으면 분할 | (사용 안 함) | 조 또는 항 단위 |

**조문형은 계층적 청킹 불필요:**
- 각 조문이 자기 완결적
- 정의 조항도 각 호가 독립적
- 일반 청킹으로 충분

### 3.4 청킹 유형별 상세 전략

#### 유형 1: 계층형 - 계층적 청킹 (전자상거래, 특수판매 지침)

**원본 JSON 구조:**
```json
{
  "번호": "가",
  "제목": "법 제2조제2호의 규정에 의한 \"통신판매\"는...",
  "내용": [
    { "본문": "...사업자의 판매에 관한 정보제공 및 소비자의 청약이..." },
    { "구분": "예시", "내용": [] },
    { "번호": "(1)", "내용": "갑은 인터넷...", "세부": [{"설명": "..."}] },
    { "번호": "(2)", "내용": "사업자 C는...", "세부": [{"설명": "..."}] },
    { "번호": "(3)", "내용": "사업자 E는...", "세부": [{"설명": "..."}] }
  ]
}
```

**청킹 구조:**
```
부모 청크 1개:
└─ "가. 통신판매 정의" 전체 (정의 + 예시 1,2,3)

자식 청크 4개:
├─ 자식 1: 도입 본문 (정의만)
├─ 자식 2: 예시 (1)
├─ 자식 3: 예시 (2)
└─ 자식 4: 예시 (3)
```

**청크 생성 원칙:**
- 예시가 2개 이상: 예시별 개별 자식 청크 생성
- 예시가 없거나 1개: 소분류 전체를 하나의 자식 청크로
- 부모는 항상 소분류 전체

---

#### 유형 2: 계층형 - ㅇ 기호 예시 (특수판매 지침)

**원본 JSON 구조:**
```json
{
  "기호": "ㅇ",
  "내용": "A신용카드사는 통신판매업자인 B여행사를 위하여...",
  "세부": [
    { "설명": "이러한 경우, A신용카드사는 통신판매업자에 해당한다." },
    { "항목": "반면에 A신용카드사가 우편물에..." },
    { "설명": "A카드사는 통신판매업자에 해당하지 아니한다." },
    { "참고": "인도에 간이판매장..." }
  ]
}
```

**청킹 구조:**
```
부모 청크:
└─ "나. 통신판매업무 수행자" 전체

자식 청크:
├─ 도입 본문
└─ ㅇ 예시 (반대 사례 + 참고 포함)
```

**특수 처리:**
- `항목` 키의 "반면에": 반대 사례로 인식 → `has_counterexample: true`
- `참고` 키: 추가 정보 → `has_reference: true`

---

#### 유형 3: 조문형 - 일반 청킹 (콘텐츠이용자 보호지침)

**원본 JSON 구조:**
```json
{
  "조": "제2조",
  "제목": "정의",
  "내용": [
    { "본문": "이 지침에서 사용하는 용어의 뜻은 다음과 같다." },
    { "호": "1", "내용": "\"콘텐츠\"란...", "세부": [] },
    { "호": "2", "내용": "\"사업자\"란...", "세부": [] },
    { "호": "5", "내용": "\"표시\"란...", "세부": [
      { "목": "가", "내용": "..." },
      { "목": "나", "내용": "..." }
    ]}
  ]
}
```

**청킹 전략:**
- **계층적 청킹 사용 안 함** (각 조문/호가 독립적)
- 조 단위 또는 호 단위로 일반 청킹
- 호가 5개 이상: 호별 분할 고려

## 4. 메타데이터 구조

### 4.1 공통 메타데이터

```json
{
  "chunk_id": "전자상거래지침_Ⅱ_2_가_예시(1)",
  "법령명": "전자상거래 등에서의 소비자보호 지침",
  "법령번호": "",
  "시행일": "2025-10-24",
  "소관부처": "공정거래위원회",
  "문서유형": "행정규칙",

  "구조유형": "계층형",  // "계층형" | "조문형"
  "대분류": "Ⅱ",
  "대분류_제목": "일반사항",
  "중분류": "2",
  "중분류_제목": "통신판매, 통신판매업자...",
  "소분류": "가",
  "소분류_제목": "법 제2조제2호의 규정에 의한...",

  "hierarchy_path": "Ⅱ. 일반사항 > 2. 통신판매 개념 > 가. 통신판매 정의 > 예시 (1)",

  "text": "...",
  "text_length": 250
}
```

### 4.2 계층적 청킹 메타데이터 (핵심!)

```json
{
  "chunk_type": "부모_청크",  // "부모_청크" | "자식_청크"
  "parent_chunk_id": null,  // 부모인 경우 null
  "자식_청크_ids": [  // 부모만 가짐
    "전자상거래지침_Ⅱ_2_가_도입",
    "전자상거래지침_Ⅱ_2_가_예시(1)",
    "전자상거래지침_Ⅱ_2_가_예시(2)",
    "전자상거래지침_Ⅱ_2_가_예시(3)"
  ],

  "is_indexed": false,  // 검색 대상 여부 (부모: false, 자식: true)
  "has_embedding": false  // 임베딩 여부 (부모: false, 자식: true)
}
```

**또는 자식 청크:**

## 6. 검색 시나리오 및 흐름

### 6.1 계층적 청킹 검색 흐름

```python
# 1. 사용자 쿼리
query = "인터넷 광고 보고 가게 가서 산 건 통신판매야?"
query_embedding = openai.embed(query)

# 2. 자식 청크만 검색 (is_indexed=true)
child_results = db.execute("""
    SELECT chunk_id, parent_chunk_id, text,
           1 - (embedding <=> %s) AS similarity
    FROM chunks
    WHERE is_indexed = true  -- 자식만!
    ORDER BY embedding <=> %s
    LIMIT 10
""", (query_embedding, query_embedding))

# 결과:
# 1. "전자상거래지침_Ⅱ_2_가_예시(1)" - 유사도 0.92 🎯
#    parent: "전자상거래지침_Ⅱ_2_가_부모"
# 2. "특수판매지침_Ⅲ_1_가_예시ㅇ" - 유사도 0.68
# 3. ...

# 3. 부모 청크 가져오기 (Top-3 자식의 부모들)
parent_ids = set(r['parent_chunk_id'] for r in child_results[:3])

parent_chunks = db.execute("""
    SELECT chunk_id, text, text_length, 자식_청크_ids
    FROM chunks
    WHERE chunk_id = ANY(%s)
""", (list(parent_ids),))

# 4. LLM에 부모 청크 전달
for parent in parent_chunks:
    context = parent['text']  # 정의 + 예시 1,2,3 전체
    answer = llm.generate(
        context=context,
        query=query,
        temperature=0.3
    )
```

## 7. 키워드 추출 전략

### 7.1 예시 청크 전용 키워드

**자동 추출 패턴:**

```python
def extract_case_keywords(text: str) -> List[str]:
    keywords = []

    # 1. 주체 패턴
    주체_패턴 = r'(소비자|사업자|판매자|판매원|[A-Z][\w]*(?:사|은|가))'
    keywords.extend(re.findall(주체_패턴, text))

    # 2. 행위 패턴
    행위_패턴 = r'(방문하여|전화하여|구매하였다|판매하였다|체결|청약|계약)'
    keywords.extend(re.findall(행위_패턴, text))

    # 3. 장소/매체 패턴
    장소_패턴 = r'(사이버몰|인터넷|가정|사업장|전화|카탈로그|우편)'
    keywords.extend(re.findall(장소_패턴, text))

    # 4. 특징 패턴
    특징_패턴 = r'(비대면|대면|직접|간접|사업장\s*외)'
    keywords.extend(re.findall(특징_패턴, text))

    return list(set(keywords))
```

### 7.2 설명(⇒) 기반 판단 결과 추출

```python
def extract_judgment(세부: List[Dict]) -> str:
    for item in 세부:
        if "설명" in item:
            text = item["설명"]

            # "~에 해당한다" 패턴
            if "해당한다" in text:
                match = re.search(r'(.+?)에?\s*해당한다', text)
                if match:
                    return match.group(1) + " 해당"

            # "~에 해당하지 않는다" 패턴
            elif "해당하지" in text or "해당되지" in text:
                match = re.search(r'(.+?)에?\s*해당하지', text)
                if match:
                    return match.group(1) + " 해당 안 함"

    return ""
```

## 9. Python 스크립트 구조

### 9.1 클래스 구조

```python
class GuideChunker:
    """행정규칙 계층적 청킹 클래스"""

    def __init__(self, guide_dir: str = "01_parsed/02_Guide"):
        self.guide_dir = Path(guide_dir)
        self.chunks = []  # 부모 + 자식 모두 저장

    def chunk_guide_file(self, file_path: Path) -> List[Dict]:
        """단일 행정규칙 파일 청킹"""
        data = json.load(open(file_path, 'r', encoding='utf-8'))

        구조유형 = self._determine_structure_type(data)

        if 구조유형 == "계층형":
            return self._chunk_hierarchical(data)
        else:  # 조문형
            return self._chunk_article_based(data)

    def _chunk_hierarchical(self, data: Dict) -> List[Dict]:
        """계층형 행정규칙 계층적 청킹"""
        chunks = []

        for 대분류_item in data["본문"]:
            for 중분류_item in 대분류_item.get("내용", []):
                for 소분류_item in 중분류_item.get("내용", []):
                    # 예시 개수 확인
                    예시_항목들 = self._extract_examples(소분류_item["내용"])

                    if len(예시_항목들) >= 2:
                        # 계층적 청킹: 부모 + 자식들
                        parent, children = self._create_hierarchical_chunks(
                            소분류_item, 대분류_item, 중분류_item
                        )
                        chunks.append(parent)
                        chunks.extend(children)
                    else:
                        # 일반 청킹: 자식만 (부모 없음)
                        chunk = self._create_section_chunk(
                            소분류_item, 대분류_item, 중분류_item
                        )
                        chunks.append(chunk)

        return chunks

    def _create_hierarchical_chunks(
        self, 소분류_item: Dict, 대분류_item: Dict, 중분류_item: Dict
    ) -> Tuple[Dict, List[Dict]]:
        """계층적 청킹: 부모 1개 + 자식 N개"""

        # 1. 부모 청크 생성 (소분류 전체)
        parent_chunk = {
            "chunk_id": self._generate_parent_id(...),
            "chunk_type": "부모_청크",
            "parent_chunk_id": null,
            "자식_청크_ids": [],  # 나중에 채움

            "text": self._build_full_section_text(소분류_item),  # 전체
            "text_length": ...,

            "is_indexed": False,  # 검색 안 함!
            "has_embedding": False,  # 임베딩 안 함!

            # 기타 메타데이터...
        }

        # 2. 자식 청크들 생성
        children = []

        # 2-1. 도입 본문 자식 (정의/개념)
        if self._has_intro_text(소분류_item):
            intro_child = self._create_intro_child(소분류_item, parent_chunk['chunk_id'])
            children.append(intro_child)

        # 2-2. 예시별 자식
        예시_항목들 = self._extract_examples(소분류_item["내용"])
        for 예시 in 예시_항목들:
            example_child = self._create_example_child(
                예시, 소분류_item, parent_chunk['chunk_id']
            )
            children.append(example_child)

        # 3. 부모에 자식 ID 연결
        parent_chunk['자식_청크_ids'] = [c['chunk_id'] for c in children]

        return parent_chunk, children

    def _create_example_child(
        self, 예시: Dict, 소분류_item: Dict, parent_id: str
    ) -> Dict:
        """예시 자식 청크 생성"""

        chunk = {
            "chunk_id": self._generate_child_id(...),
            "chunk_type": "자식_청크",
            "parent_chunk_id": parent_id,  # 부모 참조!
            "자식_청크_ids": None,

            "text": self._build_example_text(소분류_item, 예시),
            "text_length": ...,

            "is_indexed": True,  # 검색 대상!
            "has_embedding": True,  # 임베딩 생성!

            "is_example": True,
            "example_type": "번호" if "번호" in 예시 else "기호",
            "example_id": 예시.get("번호") or 예시.get("기호"),

            # 설명 추출
            "has_explanation": self._has_explanation(예시.get("세부", [])),
            "has_counterexample": self._has_counterexample(예시.get("세부", [])),
            "explanation_summary": self._extract_explanation_summary(예시.get("세부", [])),

            # 키워드 추출
            "case_keywords": self._extract_case_keywords(예시["내용"]),
            "판단결과": self._extract_judgment(예시.get("세부", [])),

            # 기타 메타데이터...
        }

        return chunk
```

## 10. 데이터베이스 스키마

### 10.1 PostgreSQL 스키마 변경

```sql
-- 기존 chunks 테이블에 계층적 청킹 컬럼 추가
ALTER TABLE chunks ADD COLUMN chunk_type TEXT;  -- "부모_청크" | "자식_청크"
ALTER TABLE chunks ADD COLUMN parent_chunk_id TEXT;  -- 부모 참조
ALTER TABLE chunks ADD COLUMN 자식_청크_ids TEXT[];  -- 자식 목록 (부모만)
ALTER TABLE chunks ADD COLUMN is_indexed BOOLEAN DEFAULT TRUE;  -- 검색 대상
ALTER TABLE chunks ADD COLUMN has_embedding BOOLEAN DEFAULT TRUE;  -- 임베딩 여부

-- 인덱스 생성
CREATE INDEX idx_parent_chunk_id ON chunks(parent_chunk_id);
CREATE INDEX idx_chunk_type ON chunks(chunk_type);
CREATE INDEX idx_is_indexed ON chunks(is_indexed) WHERE is_indexed = true;

-- 부모-자식 외래키 제약 (선택)
ALTER TABLE chunks ADD CONSTRAINT fk_parent_chunk
    FOREIGN KEY (parent_chunk_id) REFERENCES chunks(chunk_id);
```

### 10.2 검색 쿼리

```sql
-- 1. 자식 청크만 벡터 검색
SELECT
    chunk_id,
    parent_chunk_id,
    text,
    1 - (embedding <=> :query_embedding) AS similarity
FROM chunks
WHERE is_indexed = true  -- 자식만!
  AND is_example = true  -- 예시만 (선택)
ORDER BY embedding <=> :query_embedding
LIMIT 10;

-- 2. 부모 청크 가져오기
SELECT
    chunk_id,
    text,
    text_length,
    자식_청크_ids
FROM chunks
WHERE chunk_id = ANY(:parent_ids);

-- 3. 하이브리드 검색 (키워드 + 벡터)
WITH keyword_results AS (
    SELECT chunk_id, ts_rank(...) AS rank
    FROM chunks
    WHERE is_indexed = true
      AND to_tsvector('korean', text) @@ plainto_tsquery('korean', :query)
),
vector_results AS (
    SELECT chunk_id, 1 - (embedding <=> :embedding) AS similarity
    FROM chunks
    WHERE is_indexed = true
    ORDER BY embedding <=> :embedding
    LIMIT 100
)
SELECT
    c.chunk_id,
    c.parent_chunk_id,
    COALESCE(k.rank, 0) * 0.4 + COALESCE(v.similarity, 0) * 0.6 AS score
FROM chunks c
LEFT JOIN keyword_results k ON c.chunk_id = k.chunk_id
LEFT JOIN vector_results v ON c.chunk_id = v.chunk_id
WHERE k.chunk_id IS NOT NULL OR v.chunk_id IS NOT NULL
ORDER BY score DESC
LIMIT 10;
```
