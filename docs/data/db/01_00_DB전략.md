## 🎯 통합 전략 개요

### 1. 단일 테이블 vs 분리 테이블

**선택: 단일 통합 테이블 (Unified Table)**

**근거**:
- ✅ 두 데이터셋 모두 동일한 임베딩 모델 사용 (1536차원)
- ✅ 벡터 검색 시 A+B 통합 검색 필요성 (법령+사례 동시 검색)
- ✅ 단일 HNSW 인덱스로 통합 검색 성능 최적화
- ✅ 관리 복잡도 감소 (단일 스키마, 단일 인덱스)
- ✅ 하이브리드 검색 용이 (법령 근거 + 실제 사례 결합)

**대안 (분리 테이블)**:
- ❌ 통합 검색 시 UNION 연산 필요 → 성능 저하
- ❌ 두 개의 별도 벡터 인덱스 유지 필요
- ❌ 데이터셋 간 교차 검색 복잡도 증가

## 🏗️ 스키마 설계 원칙

### 1. 공통 필드 (Common Fields)

모든 청크에 공통적으로 필요한 필드:

```
- id: SERIAL PRIMARY KEY (자동 증가 ID)
- chunk_id: VARCHAR(500) UNIQUE (각 청크의 고유 식별자)
- dataset_type: VARCHAR(20) (데이터셋 구분: 'law_guide', 'case')
- text: TEXT (임베딩된 텍스트 내용)
- embedding: vector(1536) (임베딩 벡터)
- created_at: TIMESTAMP (삽입 시각)
```

### 2. 데이터셋별 메타데이터 (Dataset-Specific Metadata)

**A_law_ED_guide 전용 필드** (JSONB에 저장):
- 법령 정보: 법령명, 법령번호, 시행일, 법령유형, 모법
- 계층 정보: 편, 장, 절, 관, 조문번호, 조문제목, 항번호, 호번호
- chunk_type: 조_전체, 항_조항, 호_조항, 부모_청크, 자식_청크
- hierarchy_path: 계층 경로
- 연관 정보: 참조조문, 준용조문, 위임대상, 위임근거
- keywords: 키워드 배열
- example_cases: 예시 사례

**B_case 전용 필드** (JSONB에 저장):
- data_source: crawling, pdf
- category: 상담, 해결, 조정
- original_number: 원본 번호
- field_name: 필드명
- chunk_index, total_chunks: 청킹 정보
- metadata: 추가 메타데이터 (URL, 제목 등)

### 3. 검색 최적화 필드

빠른 필터링을 위해 자주 사용되는 필드는 별도 컬럼으로 추출:

```
- category: VARCHAR(50) (B_case의 상담/해결/조정, A_law_ED_guide는 NULL)
- law_name: VARCHAR(500) (A_law_ED_guide의 법령명, B_case는 NULL)
- chunk_type: VARCHAR(50) (A_law_ED_guide의 청크 타입, B_case는 'case')
- document_type: VARCHAR(20) (A_law_ED_guide의 문서 유형: 법률/시행령/행정규칙/별표, B_case는 NULL)
```

**document_type 필드의 중요성**:
- "해결기준"과 "해결 사례"를 명확히 구분
  - "노트북 환불 해결기준 알려줘" → dataset='law_guide', document_type='행정규칙' or '별표'
  - "노트북 환불 해결 사례 알려줘" → dataset='case', category='해결'
- 법률 문서 유형별 필터링 (법률만, 시행령만 검색 등)
- RAG 시스템의 질의 분석 정확도 향상

이를 통해:
- 메타데이터 JSONB 파싱 없이 빠른 WHERE 절 필터링
- 인덱스 생성 가능
- 통계 쿼리 성능 향상
- 사용자 의도에 정확히 매칭되는 검색

### 4. 출처 정보 필드 (Source Information)

**RAG 시스템의 신뢰도 및 추적성을 위해 출처 정보를 별도 컬럼으로 관리**:

```
- source_url: VARCHAR(1000) (B_case 크롤링 데이터 URL)
- source_file: VARCHAR(500) (B_case PDF 파일명)
- printed_page: INTEGER (B_case PDF 페이지 번호)
- source_year: INTEGER (연도 정보, 통계 및 필터링용)
```

**사용 목적**:
1. **인용 표기**: LLM 응답에 출처 자동 표시
   - 크롤링: "출처: 1372 소비자 상담센터 (URL)"
   - PDF: "출처: 2020년 소비자분쟁 해결사례집.pdf, p.48"

2. **필터링 및 검색**:
   - "2020년 이후 사례만", "특정 PDF 파일만"
   - 연도별 트렌드 분석

3. **통계 및 모니터링**:
   - PDF 파일별 사례 수
   - 크롤링 도메인별 통계
   - 연도별 데이터 분포

4. **품질 관리**:
   - 출처별 검색 품질 평가
   - 특정 출처 제외/우선순위 조정

**인덱스 전략**:
- `idx_source_url`: URL 기반 검색 (WHERE source_url LIKE ...)
- `idx_source_file`: 파일명 기반 그룹핑
- `idx_source_year`: 연도별 필터링 (WHERE source_year = 2020)
- `idx_source_file_page`: 파일+페이지 복합 인덱스 (정확한 출처 추적)

## 🔍 인덱싱 전략

### 인덱스 종류별 역할 개요

**4가지 인덱스 타입을 조합하여 최적의 검색 성능을 제공합니다:**

```
[B-tree]
  └─ 카테고리 / 연도 / 기관 필터
     (dataset_type, category, law_name, source_year, source_file 등)

[GIN - tsvector]
  └─ 법령·조문·키워드 정확 검색
     (text_tsv 컬럼, BM25 유사 키워드 검색)

[GIN - JSONB]
  └─ 메타 조건 필터
     (metadata 필드 내부 검색, keywords 배열 등)

[HNSW - pgvector]
  └─ 의미 기반 유사도
     (embedding 벡터, 코사인 유사도 검색)
```

**검색 시나리오별 인덱스 활용**:
- **하이브리드 검색**: GIN(tsvector) + HNSW(vector) → RRF 통합
- **필터링 검색**: B-tree → HNSW(vector)
- **메타데이터 검색**: GIN(JSONB) + HNSW(vector)
- **복합 조건**: B-tree + GIN + HNSW 동시 활용

### 1. 벡터 인덱스: HNSW

**선택 근거**:
- 52,452개 청크 규모에 최적 (중형 데이터셋)
- IVFFlat 대비 높은 검색 정확도
- 검색 속도: ~5ms 이내 (top-k 검색)

**설정**:
```sql
CREATE INDEX idx_embedding_hnsw ON vector_chunks
USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);
```

**파라미터 설명**:
- `m = 16`: 각 노드의 연결 수 (기본값, 균형 잡힌 성능)
- `ef_construction = 64`: 구축 시 탐색 범위 (기본값)
- `vector_cosine_ops`: 코사인 유사도 기반 검색

**대안**: IVFFlat (10만 개 이상 규모에 적합, 현재는 과도)

### 2. 메타데이터 인덱스

**GIN 인덱스** (JSONB 필드):
```sql
CREATE INDEX idx_metadata_gin ON vector_chunks USING GIN(metadata);
```

**용도**:
- JSON 내부 필드 검색 (`metadata @> '{"법령명": "민법"}'`)
- 키워드 배열 검색 (`metadata->'keywords' ? '소비자'`)

**B-tree 인덱스** (자주 필터링되는 필드):
```sql
CREATE INDEX idx_dataset_type ON vector_chunks(dataset_type);
CREATE INDEX idx_category ON vector_chunks(category);
CREATE INDEX idx_law_name ON vector_chunks(law_name);
CREATE INDEX idx_chunk_type ON vector_chunks(chunk_type);
CREATE INDEX idx_document_type ON vector_chunks(document_type);  -- 문서 유형 필터링
```

### 3. 복합 인덱스

자주 사용되는 필터 조합:
```sql
CREATE INDEX idx_dataset_category ON vector_chunks(dataset_type, category);
CREATE INDEX idx_law_chunk_type ON vector_chunks(law_name, chunk_type);
```

### 4. 전문 검색 인덱스 (Full-Text Search)

**하이브리드 검색을 위한 키워드 검색 인덱스**:

```sql
-- tsvector 컬럼 (자동 업데이트)
text_tsv tsvector

-- GIN 인덱스
CREATE INDEX idx_text_tsv ON vector_chunks USING GIN(text_tsv);
```

**용도**:
- BM25 유사 키워드 검색
- 법령명, 조항 번호 정확 매칭
- 벡터 검색과 결합하여 하이브리드 검색

## 🔀 하이브리드 검색 전략 (BM25 + 벡터 + RRF)

### 개요

**법률 도메인에서 하이브리드 검색이 필수인 이유**:
1. ✅ **정확한 키워드 매칭**: "소비자기본법 제16조" 같은 정확한 법령 검색
2. ✅ **의미 기반 검색**: "환불 거부당했어요" 같은 자연어 검색
3. ✅ **성능 향상**: 단일 검색 대비 **Recall@10 +15-20%** 개선

### 검색 파이프라인

```
사용자 쿼리: "소비자기본법 제16조 환불 거부"
         ↓
    ┌───────────────┐
    │ 쿼리 처리     │
    │ - 키워드 추출 │
    │ - 임베딩 생성 │
    └───────┬───────┘
            ↓
    ┌───────┴────────────────┐
    ↓                        ↓
┌─────────────┐      ┌──────────────┐
│ BM25 검색   │      │  벡터 검색   │
│ (키워드)    │      │  (의미)      │
│             │      │              │
│ Top 100     │      │  Top 100     │
└─────┬───────┘      └──────┬───────┘
      │                     │
      └──────────┬──────────┘
                 ↓
        ┌────────────────┐
        │  RRF 알고리즘  │
        │  (순위 융합)   │
        └────────┬───────┘
                 ↓
        ┌────────────────┐
        │  최종 Top 10   │
        │  + 출처 정보   │
        └────────────────┘
```

### RRF (Reciprocal Rank Fusion) 알고리즘

**공식**:
```
RRF_score(chunk) = Σ 1 / (k + rank_i)

여기서:
- k = 60 (RRF 상수, 일반적으로 60 사용)
- rank_i = 각 검색 방법에서의 순위
```

**예시**:
```
청크 A:
- BM25 순위: 1위 → 1/(60+1) = 0.0164
- 벡터 순위: 3위 → 1/(60+3) = 0.0159
- RRF 스코어: 0.0323

청크 B:
- BM25 순위: 5위 → 1/(60+5) = 0.0154
- 벡터 순위: 1위 → 1/(60+1) = 0.0164
- RRF 스코어: 0.0318

→ 청크 A가 최종 1위 (두 검색에서 모두 상위권)
```

### 구현 방법

**PostgreSQL 함수**:
```sql
-- 하이브리드 검색 (RRF 통합)
SELECT * FROM search_hybrid_rrf(
    '소비자기본법 제16조 환불',              -- 키워드 쿼리
    '[0.1, 0.2, ...]'::vector(1536),        -- 벡터 임베딩
    NULL,                                    -- filter_dataset
    NULL,                                    -- filter_category
    NULL,                                    -- filter_year
    10,                                      -- result_limit
    60                                       -- rrf_k (기본값)
);
```

**FastAPI에서**:
```python
@app.post("/api/search/hybrid")
async def search_hybrid(request: SearchRequest):
    # 1. 쿼리 임베딩 생성
    query_embedding = await get_embedding(request.query)

    # 2. PostgreSQL 하이브리드 검색
    results = await db.search_hybrid_rrf(
        query_text=request.query,
        query_embedding=query_embedding,
        filter_dataset=request.dataset_filter,
        filter_category=request.category_filter,
        filter_year=request.year_filter,
        result_limit=request.top_k,
        rrf_k=60
    )

    # 3. 출처 정보 포함하여 반환
    return format_search_results(results)
```

**우선순위**:
1. **A_law_ED_guide** (6,138개) - 먼저 삽입

### 2. 배치 삽입

**배치 크기**: 1,000개씩 (psycopg2 `execute_values` 사용)

```python
batch_size = 1000
for i in range(0, len(chunks), batch_size):
    batch = chunks[i:i+batch_size]
    execute_values(cursor, insert_query, batch)
    conn.commit()
```

**예상 삽입 시간**:
- A_law_ED_guide: ~2-3분 (6,138개)
- B_case: ~20-25분 (46,314개)
- **총 예상 시간**: ~25-30분

### 3. 데이터 변환

**A_law_ED_guide**:
```python
{
    "chunk_id": chunk["chunk_id"],
    "dataset_type": "law_guide",
    "text": chunk["text"],
    "embedding": chunk["embedding"],
    "law_name": chunk.get("법령명"),
    "chunk_type": chunk.get("chunk_type"),
    "category": None,
    "document_type": extract_document_type(chunk),  # '법률', '시행령', '행정규칙', '별표'
    # 출처 정보
    "source_url": None,
    "source_file": None,
    "printed_page": None,
    "source_year": extract_year(chunk.get("시행일")),  # 예: "2026-01-01" → 2026
    # 메타데이터

        "법령번호": chunk.get("법령번호"),
        "시행일": chunk.get("시행일"),
        "hierarchy_path": chunk.get("hierarchy_path"),
        "keywords": chunk.get("keywords", []),
        # ... 기타 필드
    }

# document_type 추출 함수
def extract_document_type(chunk):
    """청크에서 document_type 추출"""
    # 1. metadata에 법령유형이 있는 경우
    if chunk.get("법령유형") == "법률":
        return "법률"
    elif chunk.get("법령유형") == "시행령":
        return "시행령"

    # 2. metadata에 문서유형이 있는 경우
    if chunk.get("문서유형") == "행정규칙":
        return "행정규칙"

    # 3. chunk_id나 chunk_type에서 판단
    chunk_id = chunk.get("chunk_id", "")
    chunk_type = chunk.get("chunk_type", "")

    if "별표" in chunk_id or "별표" in chunk_type:
        return "별표"

    # 4. law_name이 있으면서 위 조건에 해당 안 되면 행정규칙
    if chunk.get("법령명"):
        return "행정규칙"

    return None
}

WITH relevant_chunks AS (
    SELECT chunk_id, metadata
    FROM vector_chunks
    WHERE dataset_type = 'law_guide' AND chunk_type = '자식_청크'
    ORDER BY embedding <=> $1::vector
    LIMIT 5

SELECT vc.chunk_id, vc.text, vc.chunk_type
FROM vector_chunks vc
WHERE vc.chunk_id IN (
    SELECT jsonb_array_elements_text(metadata->'부모_청크_id')
    FROM relevant_chunks
);
```

## 🚀 RAG 시스템 통합 전략

### 1. 검색 파이프라인 (하이브리드)

```
사용자 쿼리: "환불 거부당했을 때 소비자 권리는?"
    ↓
1. 쿼리 처리
   ├─ 키워드 추출: "환불 거부 소비자 권리"
   └─ 임베딩 생성: OpenAI API
    ↓
2. PostgreSQL 하이브리드 검색 (병렬 실행)
   ├─ BM25 검색: "환불", "거부", "소비자", "권리" 키워드 매칭
   └─ 벡터 검색: 의미적 유사도
    ↓
3. RRF 알고리즘으로 순위 통합
   └─ BM25 순위 + 벡터 순위 → 최종 순위
    ↓
4. 출처 정보 포함 Top-k 반환
   ├─ 법령: "소비자기본법 제16조 (2023)"
   ├─ 사례: "2020년 해결사례집, p.48"
   └─ URL: "https://consumer.go.kr/..."
    ↓
5. LLM 컨텍스트 생성 (출처 포함)
```

### 2. 재랭킹 전략

**RRF 기본 재랭킹** (PostgreSQL 내장):
- BM25 순위와 벡터 순위를 RRF 알고리즘으로 통합
- 추가 재랭킹 없이도 우수한 성능

**고급 재랭킹** (선택적, Python 레벨):
1. **교차 인코더** (Cross-Encoder) 재랭킹
   - 모델: `cross-encoder/ms-marco-MiniLM-L-12-v2`
   - RRF Top 20 → Cross-Encoder → Final Top 10
   - 정확도 +5-10%, 시간 +50-100ms

2. **메타데이터 부스팅**:
   - 최신 법령 우선 (source_year DESC)
   - 공식 출처 우선 (PDF > 크롤링)
   - 완전 일치 키워드 부스팅

3. **다양성 보장**:
   - MMR (Maximal Marginal Relevance)
   - 법령/사례 균형 (3:7 또는 5:5)

### 3. 출처 표기 전략 (Citation Strategy)

**RAG 응답에 출처를 자동으로 포함하여 신뢰도 향상**:
