### 5.2 구조 기반 vs 의미 기반 비교

#### 용어 정리

"의미 기반 분할"이란 용어는 실제로 두 가지 의미로 사용됩니다:

1. **넓은 의미**: 문맥을 고려한 분할
   - 문장 중간에 끊지 않기
   - 단락/문장 경계 존중
   - **RecursiveCharacterTextSplitter가 여기 속함**

2. **좁은 의미**: 임베딩 기반 의미 유사도 계산
   - 문장 간 유사도 측정
   - 주제 전환 지점 자동 감지
   - **SemanticChunker가 여기 속함**

#### 구조 기반 분할 (Structural / Rule-based)

**대표**: LangChain `RecursiveCharacterTextSplitter`

```python
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=2000,
    chunk_overlap=200,
    separators=["\n\n", "\n", ". ", "? ", "! ", " ", ""]
)
```

**작동 원리**:
```
1. 단락 구분자(\n\n)로 시도
2. 안 되면 줄바꿈(\n)으로 시도
3. 안 되면 문장 종결(마침표)로 시도
4. 안 되면 공백으로 시도
5. 최후엔 글자 단위로 분할
```

**특징**:
- ✅ 빠름 (밀리초 단위)
- ✅ 비용 없음 ($0)
- ✅ 예측 가능
- ✅ 문장 경계 보존
- ❌ 주제 전환 자동 감지 못함
- ❌ 의미 유사도 미고려

#### 의미 기반 분할 (Semantic / Embedding-based)

**대표**: LangChain `SemanticChunker` (Peak 방식)

```python
from langchain_experimental.text_splitter import SemanticChunker
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(
    model="text-embedding-3-large",
    dimensions=1536
)

splitter = SemanticChunker(
    embeddings=embeddings,
    breakpoint_threshold_type="percentile",
    breakpoint_threshold_amount=95
)
```

**작동 원리**:
```
1. 텍스트를 문장 단위로 분리
2. 각 문장의 임베딩 벡터 생성 (API 호출)
3. 연속 문장 간 코사인 유사도 계산
4. 유사도 그래프에서 급격한 하락 지점 감지
5. 봉우리(peak) 지점을 청크 경계로 선택
```

**특징**:
- ✅ 진짜 주제 전환 감지
- ✅ 의미적으로 일관된 청크
- ✅ 자연스러운 분할
- ❌ 느림 (문장당 API 호출)
- ❌ 비용 높음 ($900 for 18K docs)
- ❌ 예측 불가능

### 5.3 구조 기반으로 충분한 이유

#### 1. 법률 문서는 이미 구조화되어 있음

```python
# 조정사례의 실제 구조
{
  "case_summary": "사건 개요",        # 독립적 섹션
  "party_claims": "당사자 주장",      # 독립적 섹션
  "judgment": "판단",                # 독립적 섹션
  "decision": "결정"                 # 독립적 섹션
}
```

**각 필드 자체가 이미 의미 단위**:
- `judgment` 필드 = 법원의 판단만 포함
- 다른 주제(사건 경위, 결론)는 다른 필드에 있음
- **필드 간 주제 전환은 이미 분리됨**

#### 2. 필드 내부도 단락으로 구조화됨

법률 문서의 판단 부분은 일반적으로:

```
1. 법률 근거 제시
   - 소비자기본법 제16조...
   - 민법 제398조...

2. 사실 관계 분석
   - 본 사안에서...
   - 당사자 주장을 종합하면...

3. 법률 적용
   - 위 법률에 따르면...
   - 판례를 고려할 때...
```

**이미 단락(`\n\n`)으로 구분되어 있음**:
- 각 단락 = 논리적 블록
- RecursiveCharacterTextSplitter가 이 구조를 자동으로 존중
- 의미 손실 없음

#### 3. 실제 분할 예시

**원본 텍스트 (2,500자)**:
```
신청인은 2021년 1월 1일 피신청인과 가구 구매 계약을 체결하였다.
계약금 300만원을 지불하고 배송일을 2월 1일로 정하였다.
그러나 피신청인은 배송일을 지키지 않았고, 신청인은 2월 15일
해약을 통보하였다. 피신청인은 위약금 200만원을 요구하였다.

소비자기본법 제16조에 따르면, 소비자의 귀책사유 없는 해약의 경우
계약금에서 5%만 공제하고 환불하여야 한다.
본 사안에서 배송 지연은 피신청인의 귀책사유이므로,
신청인은 전액 환불을 받을 권리가 있다.

따라서 피신청인은 신청인에게 계약금 전액을 환불하여야 한다.
```

**구조 기반 분할 결과**:
- Chunk 1: 사건 경위 (첫 번째 단락)
- Chunk 2: 법률 근거 + 판단 (두 번째 단락)
- Chunk 3: 결론 (세 번째 단락)

**의미 기반으로 바꾸면?**:
- Chunk 1: 사건 경위
- Chunk 2: 법률 판단 + 결론 (유사도로 묶임)

**차이점**:
- 청크 경계가 약간 다름
- 하지만 **실질적 의미 차이 거의 없음**
- 법률 문서는 이미 구조가 명확하기 때문

#### 4. judgement 같은 긴 필드(3,000자)도 구조 기반으로 충분

**예시: 실제 조정사례 judgment (3,200자)**

```python
"""
□ 신청인은 이 사건 인터넷 쇼핑몰에서 매트리스 무료 체험이 가능하다는
광고를 보고 이 사건 매트리스 무료 체험을 신청한 것이라고 주장하고,
이에 대하여 피신청인은 위 인터넷 쇼핑몰 상품페이지에 29cm 매트리스는
무료 체험 대상에서 제외된다는 점을 고지하였으므로...

□ 살피건대, 이 사건 인터넷 쇼핑몰 상품페이지의 기재에 의하면
피신청인이 위 인터넷 쇼핑몰에서 판매하는 제품에 대하여 일정 기간
무료 체험을 할 수 있다고 광고한 사실...

□ 위 인정 사실 및 다음과 같은 사정을 종합하면...
"""
```

**RecursiveCharacterTextSplitter 분할 결과**:
- Chunk 1 (1,800자): 주장 부분 (첫 번째 □ 단락)
- Chunk 2 (1,400자): 사실 분석 + 결론 (나머지 □ 단락들)

**분석**:
- ✅ `□` 단락 구분자를 자동 인식
- ✅ 각 청크가 논리적 블록 완결
- ✅ 주장 → 사실 분석 → 결론 순서 유지
- ✅ **의미 손실 없음**

#### 5. 2,000자 제한이 적절함

```
평균 판단 길이: 1,500~3,000자
→ 1-2개 청크로 자연스럽게 분할
→ 단락 경계 존중
→ 의미 손실 없음
```

#### 최종 결론

**구조 기반 청킹이 충분한 이유**:
1. ✅ 법률 문서는 이미 구조화됨
2. ✅ 필드 = 의미 단위
3. ✅ 단락 = 논리적 블록
4. ✅ 2,000자 제한 = 적절한 청크 크기
5. ✅ 비용 $0 vs $900
6. ✅ 처리 시간 1분 vs 몇 시간
7. ✅ **의미 손실 없음**

**의미 기반이 필요한 경우**:
```python

```

**하지만 우리 데이터는**:
- 단락 구분 명확

- **구조 기반으로 충분**

RecursiveCharacterTextSplitter(
    chunk_size=2000,          # 최대 2,000자
    chunk_overlap=200,         # 200자 오버랩
    length_function=len,       # 문자 수 기준

    keep_separator=True

### 5.5 분할 우선순위

```

```
