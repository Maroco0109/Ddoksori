### 2.8 ContextBuilder 필드명 불일치 수정 (Phase 2-5)

**파일**: `LLM/backend/app/agents/answer_generation/context_builder.py`

**문제 원인**:
MAS Retrieval Agent들(`law_agent.py`, `criteria_agent.py`)이 반환하는 데이터 형식:
```python
{
    "chunk_id": "...",
    "content": "...",
    "metadata": {
        "law_name": "전자상거래법",      # metadata 내부!
        "article": "제17조",             # metadata 내부!
        "source_label": "소비자분쟁해결기준",
        ...
    },
    "similarity": 0.xxx
}
```

ContextBuilder가 기대한 형식 (잘못됨):
```python
law.get("law_name")   # 최상위 레벨에서 찾음 → None 반환!
law.get("article")    # 최상위 레벨에서 찾음 → None 반환!
```

**수정 내용**:
`_build_law_data()` 및 `_build_criteria_data()` 함수에서 `metadata` 딕셔너리 내부 필드도 참조하도록 수정:
```python
# 수정 전
law_name = law.get("law_name", "정보 없음")

# 수정 후
metadata = law.get("metadata") or {}
law_name = (
    law.get("law_name")
    or metadata.get("law_name")
    or law.get("doc_title")
    or "정보 없음"
)
```

**예상 효과**:
- 법령 데이터 정상 표시 (이전: "정보 없음" → 수정 후: "전자상거래법 제17조")
- 분쟁해결기준 데이터 정상 표시

### 2.9 템플릿 캐시 초기화 (Phase 2-6)

**파일**:
- `LLM/backend/app/agents/answer_generation/template_loader.py`
- `LLM/backend/app/main.py`

**문제 원인**:
`TemplateLoader`가 싱글톤 패턴으로 템플릿을 캐시하여, 서버 hot-reload 시에도 변경된 템플릿이 반영되지 않음.

**수정 내용**:

1. `template_loader.py`에 캐시 초기화 메서드 추가:
```python
@classmethod
def reload_templates(cls) -> None:
    """Force reload all templates from disk (clears singleton cache)."""
    with _template_lock:
        cls._templates = None
        cls._instance = None
```

2. `main.py` startup 이벤트에서 캐시 초기화:
```python
@app.on_event("startup")
async def startup_event():
    # 템플릿 캐시 초기화
    from app.agents.answer_generation.template_loader import TemplateLoader
    TemplateLoader.reload_templates()
```

### 2.11 LLM 프롬프트 충돌 해결 (Phase 2-8)

**파일**: `LLM/backend/app/agents/answer_generation/tools/generator.py`

**문제 원인**:
- `solution_template.md`가 `system_prompt`로 전달됨 (이미 모든 데이터와 지시사항 포함)
- 그런데 `_build_structured_prompt()`가 별도의 `user_prompt`를 생성하면서:
  - 동일한 데이터를 중복 포맷팅
  - **충돌하는 지시사항** 추가 ("각 섹션별로 정리하여 답변하고...")

**수정 내용**:
```python
# 수정 전
if user_prompt is None:
    user_prompt = self._build_structured_prompt(...)  # 항상 실행

# 수정 후
if system_prompt is None:
    # 기존 방식 (하위 호환)
    system_prompt = self._get_structured_system_prompt(...)
    if user_prompt is None:
        user_prompt = self._build_structured_prompt(...)
else:
    # 외부 템플릿 제공 시: 충돌 방지
    if user_prompt is None:
        user_prompt = "위 지시사항에 따라 답변을 생성해주세요."
```

### 2.12 Retrieval Agent 에러 수정 (Phase 2-9)

**문제 증상**:
```
[RetrievalAgent_v2:law] Error: 'NoneType' object has no attribute 'get'
[RetrievalAgent_v2:criteria] Error: 'NoneType' object has no attribute 'get'
[RetrievalMerge] laws=0, criteria=0, disputes=3, counsels=0
```

법령(law)과 분쟁해결기준(criteria) 검색이 실패하여 답변에 규정 정보가 "데이터 없음"으로 표시됨.

**원인 분석**:

1. **버그 1**: `specialized_retrievers.py`에서 정의되지 않은 `self.embed_api_url` 사용
   ```python
   # 111번 줄 (LawRetriever)
   rds = RDSInternalRetriever(self.db_config, self.embed_api_url)  # AttributeError!

   # 257번 줄 (CriteriaRetriever)
   rds = RDSInternalRetriever(self.db_config, self.embed_api_url)  # AttributeError!
   ```
   - `__init__`에서 `embed_api_url`을 `self.embed_api_url`로 저장하지 않음
   - `RDSInternalRetriever`는 두 번째 인자를 받지 않음 (deprecated 파라미터)

2. **버그 2**: `graph_mas.py`에서 `result["result"]`가 `None`일 때 처리 누락
   ```python
   # Python dict.get() 동작: 키가 존재하고 값이 None이면 기본값 무시
   result.get("result", {}).get("results", [])  # → None.get() 호출 → Error!
   ```

**수정 내용**:

**파일 1**: `LLM/backend/app/agents/retrieval/tools/specialized_retrievers.py`
```python
# 수정 전 (111번, 257번 줄)
rds = RDSInternalRetriever(self.db_config, self.embed_api_url)

# 수정 후
rds = RDSInternalRetriever(self.db_config)
```

**파일 2**: `LLM/backend/app/supervisor/graph_mas.py`
```python
# 수정 전
individual_result = {
    "documents": result.get("result", {}).get("results", []),
    ...
}

# 수정 후
result_data = result.get("result") or {}  # None이면 빈 dict 사용
individual_result = {
    "documents": result_data.get("results", []),
    ...
}
```

**예상 효과**:
- 법령 검색 정상 동작 → [규정] 섹션에 관련 법령 표시
- 분쟁해결기준 검색 정상 동작 → [규정] 섹션에 해결기준 표시
- 실패 시에도 그래프가 중단되지 않고 fallback 처리

**1. `_remove_code_block_markers()` (Step 0)**
```python
def _remove_code_block_markers(answer: str) -> str:
    """
    LLM이 답변을 마크다운 코드 블록(```)으로 감싼 경우 제거합니다.

    문제: LLM이 출력 예시의 코드 블록 형식을 그대로 따라함
    해결: 첫/마지막 줄의 ``` 제거
    """
    answer = answer.strip()

    if answer.startswith("```"):
        lines = answer.split("\n")
        if lines[0].startswith("```"):
            lines = lines[1:]
        if lines and lines[-1].strip() == "```":
            lines = lines[:-1]
        answer = "\n".join(lines)

    return answer.strip()
```

**2. `_remove_superscripts()` (Step 1)**
```python
def _remove_superscripts(answer: str) -> str:
    """
    윗첨자 숫자(¹, ², ³ 등)를 제거합니다.

    유니코드 윗첨자: ⁰, ¹, ², ³, ⁴, ⁵, ⁶, ⁷, ⁸, ⁹
    """
    superscript_map = {
        '⁰': '', '¹': '', '²': '', '³': '', '⁴': '',
        '⁵': '', '⁶': '', '⁷': '', '⁸': '', '⁹': ''
    }

    for sup, replacement in superscript_map.items():
        if sup in answer:
            answer = answer.replace(sup, replacement)

    return answer
```

**3. `_separate_bullets()` (Step 3)**
```python
def _separate_bullets(answer: str) -> str:
    """
    같은 줄에 여러 개의 bullet 포인트(●)가 있으면 분리합니다.

    변환 예:
    [유사 사례] ● 사례1 ● 사례2 → [유사 사례] ● 사례1\n\n● 사례2

    패턴: (줄바꿈이 아닌 문자)(공백)(●) → (문자)\n\n●
    """
    answer = re.sub(r'([^\n])\s+(●)', r'\1\n\n\2', answer)
    return answer
```

**기존 함수 수정**:

**4. `_fix_header_format()` - Double Newline 적용 (Step 2)**
```python
def _fix_header_format(answer: str) -> str:
    """
    섹션 헤더 뒤에 double newline(\n\n)을 추가합니다.

    ReactMarkdown은 single newline을 무시하고
    double newline만 paragraph break로 인식
    """
    headers = ["[답변 요약]", "[규정]", "[유사 사례]", "[주의 사항]", "[출처]"]

    for header in headers:
        # 패턴 1: [헤더](공백)(내용) → [헤더]\n\n(내용)
        pattern1 = re.escape(header) + r' +(.+)'
        answer = re.sub(pattern1, lambda m: header + "\n\n" + m.group(1), answer)

        # 패턴 2: [헤더]\n(내용) → [헤더]\n\n(내용)
        pattern2 = re.escape(header) + r'\n(?!\n)(.)'
        answer = re.sub(pattern2, lambda m: header + "\n\n" + m.group(1), answer)

    return answer
```

def _fix_case_numbering(answer: str) -> str:

    """
    # 변경 전
    result = header + "\n\n".join(bulleted_lines)

#### 2.18.3 핵심 문제: ReactMarkdown의 줄바꿈 처리

**문제 원인**:

ReactMarkdown(CommonMark 사양)은:
- **Single newline (`\n`)**: 무시하고 같은 paragraph로 렌더링
- **Double newline (`\n\n`)**: Paragraph break로 인식하여 줄바꿈

**증상**:
```
[유사 사례] ● 사례1 ● 사례2  ← 한 줄로 표시됨
[출처] ● 출처1 ● 출처2 ● 출처3  ← 모두 한 줄에
```

**해결**:

모든 섹션 생성/재생성 함수에서 **처음부터 double newline 사용**:

| 위치 | 변경 전 | 변경 후 |
|------|---------|---------|
| 헤더 뒤 | `[헤더]\n내용` | `[헤더]\n\n내용` |
| Bullet 사이 | `● A\n● B` | `● A\n\n● B` |
| 출처 항목 사이 | `● S1\n● S2` | `● S1\n\n● S2` |

---

#### 2.18.4 처리 순서 (6단계)

```python
def postprocess_answer(answer: str, retrieval: Optional[Dict[str, Any]] = None) -> str:
    # Step 0: 코드 블록 마커 제거 (```)
    answer = _remove_code_block_markers(answer)

    # Step 1: 윗첨자 제거 (⁰-⁹)
    answer = _remove_superscripts(answer)

    # Step 2: 헤더 형식 수정 (헤더 뒤 double newline)
    answer = _fix_header_format(answer)

    # Step 3: Bullet 분리 (같은 줄의 여러 bullet 분리)
    answer = _separate_bullets(answer)

    # Step 4: [면책 문구] → [주의 사항] 변경
    answer = answer.replace("[면책 문구]", "[주의 사항]")

    # Step 5: 유사 사례 번호 → 불릿 변경 (double newline)
    answer = _fix_case_numbering(answer)

    # Step 6: 출처 정보 보강 (double newline)
    if retrieval:
        answer = _enhance_sources(answer, retrieval)

    return answer
```

def parse_page_number(value) -> Optional[int]:
    """
    페이지 번호 파싱 (복수 페이지 처리)
    예: "43, 44" -> 43, "60-63" -> 60, "75" -> 75
    """
    if not value:
        return None

    value_str = str(value).strip()

    # 쉼표가 있으면 첫 번째 페이지만 사용 (예: "43, 44" -> "43")
    if ',' in value_str:
        value_str = value_str.split(',')[0].strip()

    # 하이픈이 있으면 시작 페이지만 사용 (예: "60-63" -> "60")
    if '-' in value_str:
        value_str = value_str.split('-')[0].strip()

    try:
        return int(value_str)
    except (ValueError, TypeError):
        return None
```

def extract_user_question_from_conversation(text: str) -> str:
    """
    대화 히스토리가 포함된 텍스트에서 실제 사용자 질문만 추출합니다.

    Frontend에서 실수로 전체 대화 히스토리를 message로 보낼 경우를 방어합니다.
    예: "[답변 요약]\n\n● 모니터...\n\n모니터 말고 전자제품으로..."
        → "모니터 말고 전자제품으로..."

    Args:
        text: 사용자 입력 (대화 히스토리가 포함되었을 수 있음)

    Returns:
        실제 사용자 질문
    """
    text = text.strip()

    # 답변 템플릿 헤더 패턴 감지
    answer_section_headers = [
        r'\[답변 요약\]',
        r'\[규정\]',
        r'\[유사 사례\]',
        r'\[주의 사항\]',
        r'\[출처\]',
        r'\[이전 대화\]',
    ]

    # 템플릿 헤더로 시작하면 마지막 유효한 문장 추출
    starts_with_template = any(re.match(header, text) for header in answer_section_headers)

    if not starts_with_template:
        return text  # 정상적인 사용자 질문

    # 마지막 줄부터 역순으로 실제 사용자 질문 찾기
    lines = text.split('\n')
    for line in reversed(lines):
        line = line.strip()
        # 템플릿/마크다운 요소 건너뛰기
        if (not line or
            line.startswith(('#', '-', '●', '[', '*', '¹', '²')) or
            re.match(r'^\d+\.', line)):
            continue

        # 의미있는 질문 (최소 5글자)
        if len(line) >= 5:
            logger.info(f"[QueryAnalysis] Extracted user question: '{line[:100]}...'")
            return line

    return text  # 실패 시 원본 반환
```

def normalize_query(query: str) -> str:
    """
    쿼리의 불필요한 접미사나 문장 부호를 제거합니다.

    NOTE: 대화 히스토리가 포함된 경우 실제 사용자 질문만 추출합니다.
    """
    # 1단계: 대화 히스토리에서 실제 사용자 질문 추출 (방어 로직)
    normalized = extract_user_question_from_conversation(query)
    normalized = normalized.strip()

    # 2단계: 불필요한 접미사 제거
    suffix_patterns = [
        r"[~해주세요|알려주세요|싶어요|인가요|할까요|있나요|있어요|될까요]$",
        r"[?？!！。\.]+$",
    ]
    for pattern in suffix_patterns:
        normalized = re.sub(pattern, "", normalized)

    return normalized.strip()
```

 * Phase 2-16: 입력 텍스트에서 불필요한 대화 히스토리 제거

const cleanUserInput = (input: string): string => {
  const text = input.trim();

  // 답변 템플릿 헤더 패턴
  const answerHeaders = [
    '[답변 요약]', '[규정]', '[유사 사례]',
    '[주의 사항]', '[출처]', '[이전 대화]',
  ];

  // 템플릿 헤더로 시작하면 마지막 유효한 문장 추출
  const startsWithTemplate = answerHeaders.some(header => text.startsWith(header));

  if (!startsWithTemplate) {
    return text; // 정상적인 사용자 질문
  }

  // 마지막 줄부터 역순으로 실제 사용자 질문 찾기
  const lines = text.split('\n');
  for (let i = lines.length - 1; i >= 0; i--) {
    const line = lines[i].trim();
    // 템플릿 요소 건너뛰기
    if (!line || line.startsWith('#') || line.startsWith('-') ||
        line.startsWith('●') || line.startsWith('[')) continue;

    if (line.length >= 5) {
      console.warn('[ChatPage] Detected conversation history, extracted:', line.substring(0, 50));
      return line;
    }
  }

  return text; // 추출 실패 시 원본 반환
};

→ 검색: 모니터 관련 사례
→ 답변: 모니터 환불 사례 제공 ✅

2차 질문: "모니터 말고 전자제품으로 더 범위를 넓혀서 유사한 사례들을 더 많이 알려줄 수 있어?"
→ 온보딩: purchase_item = "모니터" (여전히)
→ 검색: "모니터 말고 전자제품으로..." 쿼리로 검색
→ Product relevance filter: purchase_item="모니터"로 필터링
→ 답변: 여전히 모니터 사례만 제공 ❌
```

#### 2. 문제점

**문제 1: 쿼리 자체에 부정 표현 포함**
- 검색 쿼리: `"모니터 말고 전자제품으로..."`
- 벡터 검색이 "모니터" 키워드에 반응하여 모니터 문서가 상위에 랭크됨

**문제 2: Product filter가 범위 변경을 인식 못함**
- Retrieval merge에서 `purchase_item="모니터"`로 필터링
- 모니터 문서: relevance = 1.0
- 다른 전자제품 문서: relevance = 0.2 ~ 0.4 (낮음)

**문제 3: 카테고리 필터가 "모니터"를 포함함**
- PRODUCT_CATEGORY_MAP의 "전자기기"에 "모니터" 포함
- 카테고리 확장 시에도 모니터 문서가 높은 점수 유지

```python
def detect_product_scope_change(
    query: str, onboarding: Optional[Dict[str, Any]] = None
) -> Dict[str, Any]:
    """
    후속 질문에서 제품 범위 변경을 감지합니다.

    감지 패턴:
    - 부정 키워드: "말고", "제외", "대신", "다른", "아니라", "빼고"
    - 범위 확장 키워드: "범위를 넓혀", "더 넓게", "확장", "포괄"
    - 넓은 카테고리: "전자제품", "전자기기", "가전제품" 등

    Returns:
        {
            "should_ignore_product_filter": bool,  # 기존 제품 필터 무시 여부
            "expanded_category": str,              # 확장된 카테고리
            "negated_items": List[str]             # 제외하려는 품목들
        }
    """
```

**감지 로직**:
```python
# 1. 부정 키워드 체크
negation_keywords = ["말고", "제외", "대신", "다른", "아니라", "빼고"]
has_negation = any(kw in query_lower for kw in negation_keywords)

# 2. 범위 확장 키워드 체크
expansion_keywords = ["범위를 넓혀", "더 넓게", "확장", "포괄", "전반"]
has_expansion = any(kw in query_lower for kw in expansion_keywords)

# 3. 넓은 카테고리 키워드 감지
broad_categories = {
    "전자제품": "전자기기",
    "전자기기": "전자기기",
    "가전제품": "가전제품",
    # ...
}

# 4. 제외하려는 품목 추출
negated_items = []
if has_negation:
    # 온보딩에서 purchase_item 확인
    if onboarding and onboarding.get("purchase_item"):
        if purchase_item in query_lower:
            negated_items.append(purchase_item)

    # 쿼리에서 일반적인 제품명 추출 (fallback)
    for product in COMMON_PRODUCTS:
        if f"{product} 말고" in query_lower:
            negated_items.append(product)
            break

# 5. 판단
should_ignore = (has_negation or has_expansion) and expanded_category is not None
```
